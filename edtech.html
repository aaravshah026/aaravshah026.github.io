Project One

HackerRank Group Project

Background & Pain Points.

The application that we worked with is HackerRank, which is a coding practice platform designed for programmers looking for interview preparation. The main pain points are that HackerRank assumes prior knowledge, uses complicated terminology, lacks narrative around concepts, and has a non-intuitive interface.

Problem Space Target Users Pain Points

UX Research.

Our contextual inquiries found that users feel that discussions either give away answers or are not useful at all, and switching tabs for getting help can disrupt workflow. We contacted 5 people to do contextual inquiries for HackerRank. Then, we created an affinity diagram to group these issues into categories. We found that solutions included refresher modules, tracing through code, labels, and refresher questions. Users said that standalone self-reporting proficiency is not a good standalone measure of prior knowledge, and must be paired with other features.

Design Inspirations.

The design for Feature 4, Partial Code Completion, is informed by learning science principles such as scaffolding, which occurs when users receive partial support before completing full problems. This feature uses partially completed code to isolate specific skills that users need to solve the actual problem and tests to see if they have the skill. It also emphasizes procedural knowledge, since it is the strongest predictor of final grades, as stated in the course readings.

Low & Mid-Fidelity Prototyping.

The first image is the low fidelity prototype. It showed that the user would click through several partial code completion problems until they found a difficulty level with skills that they could use correctly. User Testing & Final Prototype.

The first image is from prototype 0. I took the medium fidelity prototype and converted it into a Figma design, implementing the feedback from before as well. The main feedback I got from contextual inquiries on this prototype was to fix the buttons, use more colors, use actual code lines instead of "completed code", and make the prototype more immersive. I did all of this, taking feedback from my contextual inquiries seen above along with TA and professor feedback. I got to the second image, which is the final prototype for Partial Code Completion for HackerRank.

Project Two

Individual Figma Project

Design Rationale.

I chose this topic because it’s something that I’ve had a lot of personal experience doing. I’ve also spoken at webinars about behavioral interviews for job applications. The main user flow I designed involves the user seeing the lessons on the steps to get a job, then clicking the “behavioral interview” lesson, learning the content taught in the scrollable frame, answering the quiz questions (correctly, since this is a happy path), and finishing the lesson. Key design decisions include a top-down layout and a simple navigation system to make the app user-friendly. I included component sets wherever I could to make the app as user-friendly as possible; having component sets for buttons helps the user know when they’re about to click it. I learned that a component set helps reduce the number of screens I need throughout the screens page. I don’t have to have two separate screens for when the user hovers over a button, for example. I also learned that designing in Figma is easier than it looks, and I can create a realistic clickable prototype without writing any code!
